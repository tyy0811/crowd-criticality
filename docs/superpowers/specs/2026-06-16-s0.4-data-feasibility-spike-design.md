# S0.4 — Data-feasibility spike (design)

**Date:** 2026-06-16 · **Branch:** `s0.4-data-feasibility-spike` · **Status:** design (brainstorming output; `writing-plans` + build follow).

**Companions:** `IMPLEMENTATION_PLAN.md` §2–3 (Stage 0 data-feasibility spike), `PRE_REGISTRATION.md` §4a–4c (data-source decision + validity gate), `docs/superpowers/specs/2026-06-15-s0.1-critaudit-instrument-core-design.md` (S0.1 firewalled *all* real/Polymarket/on-chain data and the feasibility spike → **S0.4**).

> This is a **feasibility instrument**, not the Stage-1 data module. Its single acceptance criterion is *a definite one-market feasibility read on source A*. It is firewalled from `src/critaudit` and is throwaway-friendly: if A fails, the spike is discarded into the reconstruction-report pivot; if A passes, its reconstruction logic is *re-built* as production `critaudit.data` (real Stage-1 work — **not** pre-counted here).

---

## 1. Why S0.4 is front-loaded — the divergent unknown

The post-S0.1 menu had three internal tracks (the real-data Gate-A.1 criterion, the Gate-A.2 demotion, the A.2 estimator redo) plus this spike. The three internal items are all **convergent**: each refines a bar that still exists (strict-GoF→LRT-based A.1; A.2→corroborative; curv vs correction-form DOF) and none forks the project. The cross-review sharpens them; it never says "this is a different project."

S0.4 is the one **divergent** unknown — the only place reality can still say *no* and force a pivot: accessible historical tick data → the **Hawkes paper**; inaccessible → the **reconstruction/validity report** (which the plan calls the *expected* pre-PhD artifact). The standing discipline is to front-load the pivot-forcer. So S0.4 runs first; the convergent methodology debt is what we bring S0.4's evidence *to*, before the pre-reg freezes. (Of the debt: the A.2 demotion and estimator redo are fully synthetic and cross-review-ready now; the real-data A.1 item waits on S0.4's actual data structure before a registered-criterion change, getting that evidence for free by sequencing after.)

## 2. Scope & definition-of-done (this push)

The core question is about **historical resolved markets**, answered by **A** (on-chain reconstruction) and **C** (paid bars) — neither calendar-bound. **B records forward**, so it cannot touch any resolved market (incl. the 2024 election); B's *outcome* is the only genuine calendar deferral.

DoD for this push:

- **Centerpiece — source A driven to a *definite* characterization** on one resolved market: either it works (with the measured numbers in §6.2) or it hits a **specific named blocker** ("A needs another session because of X" is itself a definite status).
- **C — desk rule-out**, no build (§6.5).
- **B — recorder *started*** as a cheap calendar hedge (§6.4), not a commitment to B.
- **`/prices-history` — empirically re-tested** (§6.3): measure current API reality, never inherit the documented 12h-cap claim.
- **Validity-gate thresholds — drafted** (granularity, reconciliation tolerance, wash-filter) with the **event-count floor explicitly marked S0.2-pending** (§6.6).

**Two genuine deferrals, named to keep the memo honest:** (i) the **event-count floor** (output of S0.2 synthetic estimator-stability sims; not set this push); (ii) **B's resolved-market outcome** (calendar). A "freeze-ready" verdict is *not* reachable in one session precisely because it needs both — driving for it manufactures an overclaim.

## 3. Verdict axis & criteria (load-bearing)

Reconstruction's **technical possibility is settled**: Yang & Tsang (2026) reconstructed Polymarket on-chain trade fills for the resolved 2024 election. So "A feasible?" is **not** "is reconstruction possible" but **"is it accessible to *this project* at acceptable cost/effort in the pre-PhD window."** Frame every verdict on that axis.

- **A feasible** ≙ the target market is reconstructed; its *resolvable* kernel timescale clears the synthetic quantization threshold (§6.1); and settled notional reconciles within tolerance of the **pinned expected ratio** (§6.2). → Stage-1 Hawkes proceeds.
- **A blocked** ≙ a **named access-or-cost blocker only** (e.g., "free Dune credits insufficient at project scale; paid tier ≈ $X needed," "neg-risk decoding is N days of engineering beyond the window"). → reconstruction-report pivot, with B's recorder already collecting forward.
- **"Technically infeasible" is not an available verdict** — it would contradict the cited method. An access-route artifact (raw-RPC range cap, free-tier rate limit) must **never** be written as "A blocked"; that is a wrong-tool signal (§6.2), not a finding.

These criteria are written into `RUN.md` and the memo template (§6.6) so an offsite executor cannot misclassify an access artifact as a substantive negative.

## 6. Components

### 6.1 Synthetic quantization-bias sweep — `quantization_sweep.py` (the granularity verdict, synthetic half)

"Does 2 s block-time quantization bias `n̂`?" is a **bias question**, answered on **synthetic ground truth**, not by eyeballing descriptives (tie fraction / median gap *suggest* but do not *establish* the bias, and this is the single number gating whether the whole reconstruction path is usable for `n` — exactly where the project's own bar must not quietly drop).

- Reuse the S0.1 Hawkes generator + MLE. Sweep kernel timescale `1/β` across a range spanning above and below 2 s; for each, generate events, **round timestamps to a 2 s grid**, refit `n̂` (**point estimates only — no profile CIs**).
- Report `|n̂ − n|` vs timescale and the **`T_threshold`** at which the bias breaches a stated bar.
- **Smoke gate:** the *un-quantized* refit must recover planted `n` within tolerance before the quantized comparison is trusted (pre-flight check #5 analog).
- **Exponential kernel is a valid proxy** despite markets being long-memory: quantization damage is a **short-lag** phenomenon (the long-memory tail lives at long lags where 2 s is harmless), and the exponential at the right timescale carries the short-lag mass.
- **Chicken-and-egg dissolution:** framed as **resolvable vs unresolvable**. The sweep gives `T_threshold`; the market's resolvable timescale (§6.2) is compared to it; any **sub-2 s component is flagged unmeasurable**, never assumed benign.

### 6.2 Source-A reconstruction — `dune_orderfilled.sql` + `reconstruct.py` (centerpiece)

**Market sequence — clean binary first, election/neg-risk second.** Because granularity is now answered synthetically, the high-rate election is no longer needed to stress-test it. Lead with a clean high-volume **binary** market (single Yes/No, standard CTF Exchange) to isolate the **base-path** verdict (does reconstruction work at all). Then the **2024 election** — almost certainly a multi-candidate **neg-risk** market (→ the **Neg-Risk CTF Exchange**, a different decoded contract; confirmed at run time, §10) — as an **isolated confound test**, so neg-risk cost lands as a *separate* finding ("base path works; neg-risk adds X"). Both are needed for the project verdict (the election is the headline data): binary-*then*-election, not instead-of.

**Pipeline:** Gamma/CLOB API (market metadata → condition ID, the two ERC-1155 `clobTokenIds`, reported `volume`) → **Dune SQL** over decoded `OrderFilled`/`OrdersMatched` for those token IDs across the market lifetime (block time, maker/taker amounts, price, token id, tx) → `reconstruct.py` (parse → sorted **trade-arrival point process** with size/price marks, **shaped to `critaudit.types`**; compute settled notional) → `validity_gate.py`.

**The three measured numbers that decide it:**
1. **Granularity adequacy** — the market's **resolvable kernel timescale** vs `T_threshold` (§6.1), with same-block ties / sub-2 s mass flagged. (Ties concentrate in the *bursts* that carry the self-excitation signal, so the average gap is the wrong lens.)
2. **Reconciliation** — settled notional vs Polymarket reported volume, **within tolerance of a pinned expected ratio** (see below).
3. **Event count** — recorded against the floor, but the floor is S0.2-pending, so this is reported, not gated.

**Pin the expected reconciliation ratio *before* setting tolerance.** Settled notional ≠ reported volume a priori: maker/taker fills can **double-count** (one match may emit two `OrderFilled`; `OrdersMatched` emits once), **fees** open a gap, and Polymarket "volume" may not be the same object as settled USDC notional. The pinned relationship (which event to sum / one-side-vs-both, fee treatment, what "volume" counts) is written into the memo, and the tolerance is set **around that expected ratio**, not naively around 1.0 — otherwise a units artifact reads as a reconstruction error and inverts the check (pre-flight #4 in miniature).

**Design-around-don't-assert.** Exact decoded table/event names, the maker/taker→price/size math and **decimals** (USDC 6; share units), and the standard-vs-Neg-Risk exchange are **confirmed against the live Dune schema at execution** — not baked in as if known. The spike *discovers* them (§10).

### 6.3 `/prices-history` re-test — `prices_history_probe.py` (offsite, runs first)

Query the public CLOB `/prices-history` (+ Gamma) for the chosen resolved binary market at the finest interval offered; record the **actual returned timestamp spacing**. Verdict line: "public-API max resolution for resolved markets = X (documented: 12 h)." A surprise (finer data now served) is flagged loud — it could simplify or collapse the spike. This is the "measure current reality, don't inherit the stale claim" step (could otherwise *falsely kill* or *falsely greenlight* the project).

### 6.4 Source-B recorder — `ws_recorder.py` (offsite, forward calendar hedge)

Select a basket of **open short-horizon** markets (resolves ≤ ~30 d, high recent trade rate) via Gamma; subscribe to the CLOB WebSocket market channel for their token IDs; append every trade message + a **local high-resolution receive timestamp** (sub-second — finer than A's 2 s, though forward-only) to **append-only JSONL**; reconnect/backoff, gap logging. The agent **starts** it this session (the only item with a calendar clock — starting now is the cheap hedge). It runs forward regardless of the A verdict, as the reconstruction-report path's input. **Capture only — no analysis.**

### 6.5 Source-C desk rule-out (written, no code)

Memo section: C yields minute **bars, not trades** (a price-move process — a *different* branching object than trade-arrivals, so a different `n`), **Aug-2025 onward only** (misses the 2024 election), and costs money. → **caveated fallback only**, documented and set aside.

### 6.6 Validity-gate thresholds (drafted) + `RUN.md` + memo

Drafted thresholds with provenance:
- **Granularity:** PASS iff resolvable `T ≥ T_threshold` (synthetic-calibrated, §6.1), sub-2 s component flagged. (Replaces a hand-set "finer than [fill]".)
- **Reconciliation tolerance:** set around the **pinned expected ratio** (§6.2); the pinning is a prerequisite step recorded in the memo.
- **Wash-filter:** the spike only confirms the data carries the **maker/taker addresses** needed to apply the filter; the filter (self-trade / same-address-both-sides) + the filtered-vs-unfiltered `n` delta is **Stage-1**.
- **Event-count floor:** **S0.2-pending** — recorded against, not set.

`RUN.md` (handoff for the network-capable agent): prerequisites (free Dune key); run order (sweep [local] → prices-probe → **binary** recon → **election/neg-risk** recon → reconcile → gate → start recorder); exact commands; capture list; and **the §3 verdict criteria verbatim** (feasible = binary reconstructed + `T ≥ T_threshold` + reconciliation within tolerance-of-expected-ratio; blocked = named access/cost blocker ONLY; neg-risk cost = separate finding; access-route artifacts are never "blocked").

Memo (`results/s0.4_feasibility/2026-06-16_memo.md`): the definite core verdict; the `T_threshold` curve; the binary result (+ election/neg-risk as a separate finding); reconciliation vs the pinned ratio; the `/prices-history` measurement; the drafted thresholds; and the two named deferrals.

## 7. Testing & error-handling

**Testing — pure-logic kernels only (no gold-plating):**
- `reconstruct.py` parse/notional math against a tiny hand-built `OrderFilled` fixture with known timestamps/notional — catches the units bugs that would silently corrupt reconciliation (USDC 6-decimals, maker/taker→price, double-count dedupe). Pre-flight checks #1/#3 in miniature.
- `validity_gate.py` verdict logic against two fixtures (a feasible case; an access-blocked case) — proves the criteria are encoded correctly **and** that an access/cost limit cannot be encoded as "infeasible."
- `quantization_sweep.py` self-validates (un-quantized recovers planted `n`); run locally to confirm a sensible monotone bias curve.
- **Explicitly out:** network-call tests, multi-market generality, production error paths.

**Error-handling — minimal but honest:** scripts **fail loud with the reason** so blockers are *named* ("Dune returned 0 rows for token IDs X → check exchange / neg-risk variant"). The **access/cost-≠-infeasible** rule is encoded in both `RUN.md` and the gate. Recorder: reconnect/backoff, append-only, gap log.

## 8. Deliverables & exit

- **Decision memo** (`results/s0.4_feasibility/`) with the definite core verdict and all evidence above.
- **B's recorder running** on the user's machine.
- **Fork:** A feasible → Stage-1 Hawkes proceeds, reconstruction graduates to a production `critaudit.data` module (real Stage-1 work, not pre-counted). A blocked → reconstruction-report pivot, B already collecting forward.
- **Hand-off to the convergent debt:** the memo's data-structure findings (granularity, event object, reconciliation) become the evidence the real-data Gate-A.1 item is decided against at cross-review, before the pre-reg freezes.

## 9. Non-goals / scope-out

- No production `critaudit.data` module, no committed package interface (type-*shaping* only).
- No multi-market generality, no production error paths, no comprehensive test suite.
- No event-count floor (S0.2). No B resolved-market outcome (calendar). No wash-filter implementation (Stage-1).
- No raw-RPC reconstruction path unless the indexer demonstrably lacks the needed granularity (it carries ~2 s block-time); a raw-RPC range cap / rate limit is a wrong-tool artifact, not a finding.
- No freeze of any pre-reg `[fill]`; the spike *drafts* thresholds and feeds cross-review.
